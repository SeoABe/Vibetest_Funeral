# PLAN.md 평가 및 발전 방안

## 총평

현재 `PLAN.md`는 모바일 부고장 서비스를 빠르게 구현하기 위한 1차 개발 계획서로서, 핵심 기능과 기술 스택이 명확하다. 공개 부고 페이지, 카카오 공유, 지도, 조의금 계좌, 방명록, 유언장, 관리자 페이지까지 주요 기능 범위가 잘 잡혀 있다.

다만 실제 서비스로 운영하려면 단순 기능 구현보다 더 중요한 영역이 있다. 부고 서비스는 감정적으로 민감하고, 개인정보와 금융정보를 다루며, 악성 작성·무단 열람·검색 노출·계좌 오남용 같은 위험이 있다. 따라서 UX, 보안, 개인정보보호, 운영 정책, 데이터 수명주기, 장애 대응, MVP 범위 조정이 보강되어야 한다.

## 강점

### 1. 기능 범위가 직관적이다

유족이 부고를 만들고, 링크로 공유하고, 조문객이 정보를 확인하는 기본 흐름이 잘 정의되어 있다. 부고 서비스에서 기대되는 핵심 요소인 고인 정보, 장례 일정, 위치, 조의금 계좌, 방명록이 모두 포함되어 있다.

### 2. 모바일 중심 방향이 적절하다

부고 링크는 카카오톡 등 모바일 메신저로 전달되는 경우가 많다. 최대 너비 480px, 단일 스크롤, 카카오 공유 SDK, 지도 앱 외부 링크 등은 사용 맥락과 잘 맞는다.

### 3. 기술 스택이 현실적이다

Next.js, TypeScript, Tailwind CSS, Prisma, Supabase, Vercel 조합은 1인 개발 또는 소규모 MVP에 적합하다. 빠르게 만들고 배포하기 좋으며, 관리자 페이지와 공개 페이지를 하나의 앱에서 다루기도 쉽다.

### 4. 방명록 승인 흐름이 포함되어 있다

장례 관련 공개 페이지는 스팸, 장난성 댓글, 악의적 메시지에 취약하다. 작성 즉시 공개가 아니라 관리자 승인 후 공개하는 방향은 안전한 기본값이다.

### 5. 유언장 보안 요구를 인식하고 있다

비밀번호 해시, AES 암호화, rate limiting을 계획에 포함한 점은 좋다. 다만 유언장 기능은 민감도가 매우 높으므로 현재 계획보다 더 엄격한 제품 판단이 필요하다.

## 주요 리스크

### 4. 조의금 계좌 공개는 악용 가능성이 있다

계좌번호를 공개 페이지에 노출하면 스크래핑, 피싱, 계좌 도용성 문의, 잘못된 송금 유도 문제가 생길 수 있다.

보강 방안:

- 계좌번호 전체 노출 대신 탭/버튼 후 표시
- 복사 시 계좌주와 은행명을 함께 표시
- 계좌 정보 변경 이력 저장
- 관리자 수정 시 재확인 단계 추가
- 조의금 송금은 외부 금융거래이며 서비스가 송금을 대행하지 않는다는 고지

## UX 평가

### 유족 UX

현재 계획은 개발자 관점의 기능 목록은 충분하지만, 유족이 실제로 부고를 만들 때 필요한 입력 흐름이 더 구체화되어야 한다.

개선 제안:

- 부고 생성 폼을 단계형으로 구성
- 필수 입력과 선택 입력을 명확히 분리
- 장례식장 검색 후 주소·좌표 자동 입력
- 날짜/시간 입력은 한국 장례 관례에 맞는 라벨 사용
- 작성 중 미리보기 제공
- 카카오톡 공유 전 미리보기 확인
- 임시저장 기능 제공

권장 생성 단계:

1. 고인 정보
2. 상주 및 가족 정보
3. 장례 일정
4. 장례식장 위치
5. 조의금 계좌
6. 방명록/공개 설정
7. 최종 미리보기 및 공유

### 조문객 UX

조문객은 빠르게 정보를 확인하고 이동하거나 송금하거나 메시지를 남기는 목적이 강하다. 페이지 상단에서 핵심 정보를 즉시 보여주는 것이 중요하다.

개선 제안:

- 상단에 발인일, 장례식장, 대표 상주를 요약 표시
- 지도, 계좌, 방명록으로 바로 이동하는 고정 액션 제공
- 장례식장 주소 복사와 지도 앱 열기 버튼을 명확히 분리
- 계좌 복사 후 은행/예금주/번호가 함께 복사되는 옵션 검토
- 고령 사용자도 볼 수 있도록 글자 크기와 대비 강화

## 기술 설계 평가

### Next.js App Router

공개 페이지는 SEO보다 공유 미리보기와 빠른 렌더링이 중요하다. 동적 라우트, 서버 컴포넌트, 메타데이터 생성 기능을 잘 활용하면 적합하다.

보강 제안:

- `generateMetadata`에서 부고별 OG 메타 생성
- 공개 페이지는 캐시 전략을 명확히 정의
- 관리자 페이지와 공개 페이지의 레이아웃 분리
- 서버 액션과 API route의 역할 구분

### 데이터 모델

현재 모델은 MVP로는 가능하지만, Json 필드가 많아 운영·검색·검증·마이그레이션이 어려워질 수 있다.

개선 제안:

- `familyList`, `accounts`, `visitation`은 초기에 Json으로 시작 가능
- 장기적으로 `FamilyMember`, `CondolenceAccount`, `FuneralSchedule` 모델 분리 검토
- `updatedAt` 추가
- `deletedAt` 또는 soft delete 검토
- `expiresAt` 추가
- `publishedAt` 추가
- `viewCount`는 필요 시 별도 이벤트 테이블로 분리

예상 추가 모델:

- `User`
- `AuditLog`
- `FuneralAccessLog`
- `CondolenceAccount`
- `FamilyMember`
- `GuestbookReport`

### 이미지 업로드

계획에는 고인 사진 URL만 있지만, 실제 서비스에서는 업로드 흐름이 필요하다.

결정해야 할 항목:

- Supabase Storage 사용 여부
- 업로드 파일 크기 제한
- 이미지 리사이징
- 기본 추모 이미지 제공
- 삭제 시 스토리지 파일 정리
- 부적절 이미지 신고/삭제 정책

### 지도 연동

카카오맵 API 사용은 적절하다. 다만 장례식장 주소 입력을 직접 받으면 좌표 오류가 발생할 수 있다.

개선 제안:

- 주소 검색 API 또는 장소 검색 API로 장례식장 선택
- 좌표 직접 수정 옵션은 관리자용으로만 제공
- 카카오맵, 네이버지도, 티맵 링크를 선택적으로 제공
- 모바일에서 앱 미설치 시 웹 링크 fallback 확인

### 방명록

방명록은 스팸과 악성 문구의 주요 진입점이다.

권장:

- 최대 글자 수 제한
- 금칙어 또는 신고 기능
- IP 또는 fingerprint 기반 rate limit
- 관리자 승인 전 비공개 유지
- 관리자 삭제 사유 기록

## 법무·정책 관점

실제 상용 서비스라면 아래 문서가 필요하다.

- 이용약관
- 개인정보처리방침
- 개인정보 수집·이용 동의
- 방명록 작성자 개인정보 안내
- 조의금 계좌 정보 고지
- 유언장 또는 남기는 글의 법적 효력 부인 고지
- 신고 및 삭제 요청 절차
- 서비스 보관 기간 및 파기 정책

특히 `유언장`이라는 표현은 법적 오해를 만들 수 있으므로, 초기에는 "비공개 추모글" 또는 "가족에게 남긴 글" 정도로 제품명을 조정하는 편이 안전하다.

## 운영 관점

### 운영자가 필요로 할 기능

- 부고 검색
- 공개/비공개 필터
- 방명록 승인 대기 목록
- 최근 수정 이력
- 만료 예정 부고 목록
- 계좌정보 변경 이력
- 신고된 방명록 관리
- 데이터 내보내기

### 장애 대응

부고 서비스는 특정 기간에 접속이 집중될 수 있다. 공유 직후 트래픽이 몰릴 수 있으므로 공개 페이지 성능이 중요하다.

권장:

- 공개 페이지 정적/캐시 전략 검토
- 이미지 최적화
- DB connection pooling
- Supabase/Vercel region 확인
- 에러 로깅 도입
- 관리자 작업 실패 시 재시도 UX 제공

## MVP 재정의

현재 계획의 모든 기능을 8~12일 안에 안정적으로 만들기는 가능할 수 있지만, 민감 서비스 기준으로는 검증 시간이 부족하다. MVP는 위험도가 낮고 가치가 큰 기능부터 좁히는 것이 좋다.

## 접근성 평가

장례 서비스는 다양한 연령대가 사용한다. 접근성은 선택이 아니라 핵심 품질에 가깝다.

개선 제안:

- 본문 최소 16px 이상
- 충분한 명도 대비
- 버튼 터치 영역 최소 44px
- 링크와 버튼의 시각적 구분
- 스크린리더용 라벨
- 토스트 알림 외에도 복사 성공 상태를 텍스트로 제공
- 애니메이션 감소 설정 대응

## 성능 평가

공개 부고 페이지는 빠르게 떠야 한다. 카카오톡 인앱 브라우저에서 느리면 체감 품질이 크게 떨어진다.

개선 제안:

- 첫 화면에 필요한 데이터만 우선 로드
- 지도는 지연 로딩
- 방명록은 별도 페이징 로드
- 고인 사진 최적화
- 폰트 로딩 전략 점검
- OG 이미지 생성은 캐시 적용

## 데이터 보관과 삭제

부고는 영구 보관이 필요한 경우도 있지만, 일정 기간 후 비공개를 원하는 경우도 많다.

권장 정책:

- 기본 공개 기간 설정 가능
- 유족 요청 시 즉시 비공개
- 완전 삭제와 비공개 보관 분리
- 방명록 데이터 내보내기
- 계좌정보는 장례 종료 후 자동 숨김 옵션
- 삭제된 부고의 이미지 파일 정리

### 낮은 리스크, 높은 가치

- 장례식장 장소 검색
- 카카오톡 공유 미리보기
- QR 코드 생성
- 인쇄용 부고 이미지 생성
- 계좌정보 장례 종료 후 자동 숨김
- 방명록 다운로드
- 대표 상주에게 공유 링크 복사 버튼

### 중간 리스크
- 조회 통계
- 추모 사진첩
- 장례 후 추모 페이지 전환

### 높은 리스크
- 온라인 조의금 결제
- 법적 유언장 보관
- 본인인증 기반 가족 권한 관리
- 공개 검색 가능한 추모관
